## tickets/ (PM ↔ DESIGN ↔ DEV 협업 시스템)

목적: 티켓 1장이 **PM 요구사항 + DESIGN 핸드오프 + DEV 구현/검증**을 한 곳에서 연결하는 협업 인터페이스가 된다.

---

## 1) 티켓이 곧 인터페이스(SSOT)
- 요구사항/결정/디자인/구현/검증/링크는 **티켓에 누적**한다.
- 기존 내용 삭제 금지(append만).

---

## 2) 파일 규칙
- **파일명**: `T-XXX__short_slug.md`
- **템플릿**: `agents/planning/tickets/00_TICKET_TEMPLATE.md`

---

## 3) 역할별 책임(필수)
### PM
- 티켓 생성(목적/범위/요구사항/AC) + 결재/회의 링크 첨부
- 작업 완료 후 `agents/worklogs/PM.md` 업데이트
- **금지**: 코드 수정/개발 진행

### DESIGN
- 티켓의 DESIGN 섹션에 UX/UI/카피/토큰/산출물 링크/핸드오프 작성
- 작업 완료 후 `agents/worklogs/DESIGN.md` 업데이트

### DEV
- 티켓의 DEV 섹션에 기술 설계/변경 범위/테스트 플랜 작성
- 구현 중/완료 후 티켓 진행 로그 업데이트
- 작업 완료 후 `agents/worklogs/DEV.md` 업데이트

---

## 4) 완료 게이트(강제)
- 티켓 완료(Done) 조건:
  - PM: AC가 명확하고 변경/결정 근거 링크가 있음
  - DESIGN: 산출물 링크 + 핸드오프가 있음(해당되는 티켓만)
  - DEV: 테스트/검증 + 산출물 링크(커밋/PR/스크린샷)가 있음
  - 각 역할: 본인 작업로그 업데이트 완료(미작성=미통과)

---

## 5) 운영 팁
- DEV 질문은 `Q/A & 결정 로그`에 남기고, PM/디자이너가 답변을 같은 곳에 누적한다.
- 스코프 변경이 필요하면 “결재 게이트” 규칙을 따른다.
